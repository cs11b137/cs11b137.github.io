几个 AI 小项目的复盘：什么值得长期维护

过去一两年，陆陆续续做了不少和 AI 相关的小项目。
有些上线一两个星期就被我自己放弃，有些反而默默地用到了现在。

趁着空档，把这几类小项目复盘一下：

  • 哪些值得长期维护
  • 哪些适合当一次性实验
  • 哪些在想法层面很好，但现实里难以持续

1. 个人知识库 + AI 搜索：最常用，也最值维护

1.1 做了什么

这个就是前面写过的：

  • 把博客 / 笔记内容切分、向量化
  • 用向量检索 + 大模型做一个「只基于自己内容」的问答

1.2 为什么一直在用？

  1. 明确解决了一个长期痛点

    • 「我之前好像写过这个，但忘了在哪里」
    • 知识库 + AI 搜索给了一个非常直接的解法

  2. 需求稳定，维护成本低

    • 内容只会增加，不会经常变来变去
    • 系统结构相对简单，坏了问题也好排

  3. 使用频率高且自然

    • 写新东西前想查旧文章时会用
    • 工作中碰到类似问题也会顺手问一下

结论：这是那种一旦习惯了就离不开的工具，非常值得长期维护。

2. AI 写作助手：从独立产品变成「一个流程中的环节」

2.1 一开始怎么做的？

最早我想做一个「写作工作台」：

  • 在线编辑器 + 大纲生成 + 自动扩写
  • 一键生成摘要、标签、社交媒体文案

结果做着做着发现：

  • 编辑器永远写不赢现有工具（例如本地 Markdown 工具）
  • 真要写东西时，我还是回到习惯的环境里去写

2.2 后来怎么调整？

后来干脆把它拆散：

  • 保留「提示词模板 + 一键调用模型」部分
  • 不再做自己的编辑器
  • 把这些步骤融到现有写作流程里（比如借助现有工具的插件、脚本）

2.3 现在的状态

现在的 AI 写作部分更像是：

一些我常用的 Prompt + 一些自动脚本，散落在不同工具里。

结论：

  • 「完整写作平台」这种产品形态对我来说不成立
  • 但「把 AI 用在写作不同阶段的小工具」是值得长期打磨的

3. AI Code Review 机器人：有用，但需要控制预期

3.1 做了什么

  • 在 CI 流水线里接入大模型
  • 对 PR 的 diff 做分析，给出一些风险提示和建议
  • 通过机器人账号回到代码托管平台（比如 GitHub）评论

3.2 使用体验

优点：

  • 对一些边界情况、错误处理、重复代码有不错的发现能力
  • 在人类 Reviewer 忙不过来的时候，当个辅助眼睛

缺点：

  • 如果 Prompt 不收敛，会变成「话很多但不太有用」
  • 某些建议过于保守，容易导致「任何改动都被说有风险」

3.3 是否值得长期维护？

我给的结论是：值得，但要有边界感。

  • 它适合当「辅助 reviewer」，而不是「审核人」
  • 团队要有能力不断调整 Prompt 和过滤规则
  • 不要指望它解决所有质量问题

4. 一些「很酷但荒废很快」的项目

4.1 自动生成页面 Demo / Playground

当时的想法是：

输入一段需求描述，自动生成一个前端 Demo 页面。

结果：

  • 开发体验很炫，（尤其是录视频演示的时候）
  • 真要落地到工作里的时候，大家仍然倾向于自己写代码

核心原因：

  • 生成的代码很难直接合入生产项目
  • 还要花时间「手动迁移」到现有架构、组件库、状态管理体系里

4.2 聊天式技术问答机器人

最一开始很兴奋地：

  • 接入大模型做一个「专门回答前端问题」的小机器人
  • 还给它加了一些人格设定

但没多久就发现：

  • 这个和直接用通用大模型相比，并没有明显优势
  • 反而多了一层维护负担（要调参数、要管日志）

最后回归到：

直接用主流模型 + 自己写好的提示词就够了。

5. 复盘：什么项目值得长期维护？

从这几个小项目的生死来看，有几点共性：

  1. 能融入日常工作流的更容易活下来

    • 个人知识库搜索自然融入「查旧东西」
    • 写作辅助融入「写博客」
    • AI Review 融入「提 PR」

  2. 不改变主工具，只在边上加「外挂」更稳

    • 不自己造编辑器、IDE，而是做插件和脚本
    • 不重做任务管理，而是在现有系统上叠一层辅助

  3. 能解决长期问题，而不是满足一时好奇心

    • 「找回自己写过的东西」是长期问题
    • 「让写 Demo 看起来很炫」则更像短期兴奋点

  4. 维护成本必须低

    • 依赖越少越好
    • 有问题时能快速定位并修复

6. 下一步想做什么？

基于这次复盘，我对未来想做的小项目有几个倾向：

  1. 多考虑「和个人网站 / 知识库 / 日常开发」的结合
  2. 少做「全新大平台」，多做「小工具 + 脚本 + 插件」
  3. 模型调用逻辑尽量统一，避免到处复制粘贴

简单说就是：

少造轮子，多造「轮子助手」。

7. 小结

回顾这些 AI 小项目，会发现：

  • 真正留下来的，不一定是技术上最炫的那个
  • 而是长期重复使用成本最低、给自己带来真实帮助的那个

这也是我现在看 Side Project 时更在意的一点：
不是问「能不能做」，而是问：

「一年之后，我还有没有动力点开它？」
