金融级前端系统的架构约束与设计取舍

背景

在“金融级”业务系统中，前端架构常常被一组约束牵引：

  • 稳定性与可预测性优先
  • 安全与隐私默认开启
  • 变更可追溯、可回滚
  • 权限边界清晰，最小化误操作

这意味着很多场景下，架构目标并不是“最酷/最先进”，而是：

长期可控、可演进、可治理。

1. 架构目标：把不确定性关在笼子里

可控性的体现

  • 功能开关（feature flag）：按用户/角色/灰度启用
  • 发布回滚：前端资源与配置可快速回退
  • 观测能力：错误、性能、关键链路有明确指标

不确定性的来源

  • 动态路由/动态渲染过度导致行为难预测
  • 依赖链复杂导致升级风险上升
  • 关键数据在多处重复维护导致一致性问题

2. 权限体系：从“页面权限”到“动作权限”

常见的权限粒度：

  1. 页面级：能不能进入页面
  2. 组件级：能不能看到某个模块
  3. 动作级：能不能点击某个按钮（提交/审批/导出）

推荐以“动作权限”作为最小单元：

{
  "resource": "investor:profile",
  "actions": ["view", "edit", "submit", "export"]
}

并在前端形成统一能力：

  • 路由守卫：页面进入校验
  • 指令/组件：动作级权限控制（按钮/入口）
  • 请求层：对敏感接口做二次校验（防止绕过 UI）

3. 网络层：把“稳定”变成默认行为

前端网络层建议统一封装：

  • 超时与重试策略（对幂等接口可重试）
  • 统一错误码映射（给用户可理解提示）
  • 会话失效处理（单点登出/刷新凭证）
  • 请求幂等 key（防止重复提交）

示例（策略示意）：

请求发出 -> 超时(8s) -> 可重试(最多1次) -> 失败提示(可操作)

4. 状态管理：把“关键状态”集中

金融系统常见关键状态：

  • 当前流程步骤（可恢复）
  • 登录会话与权限
  • 关键查询条件与筛选
  • 字典/枚举（全局一致）

建议：

  • 全局状态只放“跨页面共享且重要”的内容
  • 页面状态尽量本地化（避免全局污染）
  • 对流程类页面使用明确的状态模型（便于恢复与测试）

5. 组件体系：一致性比“组件多”更重要

组件体系建设建议关注：

  • 表单一致性：校验、错误提示、布局
  • 表格一致性：分页、排序、导出、列配置
  • 弹窗一致性：确认/取消、二次确认、危险操作

在金融业务中，“一致性”带来的收益很大：

  • 降低误操作
  • 降低培训成本
  • 降低维护成本（同类问题一次修复全站受益）

6. 可观测性：让问题“能被定位”

前端观测建议至少覆盖三类：

  • 错误：JS 异常、资源加载失败、接口失败
  • 性能：首屏、路由切换耗时、长任务
  • 业务链路：关键步骤成功率、失败原因分布

并做到：

  • 关键链路有 traceId（前后端联动定位）
  • 日志脱敏（默认不记录敏感字段）
  • 错误聚合去重（避免“海量重复”）

7. 发布与治理：把变更风险降下来

推荐实践：

  • 灰度发布：按角色/比例启用
  • 配置中心：关键开关与文案可配置（需治理）
  • 依赖升级策略：小步升级、范围回归、可回退

对“基础设施级”的变更（路由、鉴权、网络层）建议：

  • 先在低风险页面落地
  • 打开监控与告警阈值
  • 完成一轮回归后再扩展

总结

金融级前端架构的难点不在技术实现，而在约束下的取舍：

  • 稳定性与可预测性优先
  • 权限与状态边界清晰
  • 网络层与观测能力成为基础设施
  • 发布治理（灰度/回滚/可追溯）是常态能力

当“可控”成为默认，系统才能在长期迭代中保持速度与安全的平衡。
