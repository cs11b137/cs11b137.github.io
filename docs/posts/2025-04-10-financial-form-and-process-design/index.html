背景

金融业务系统常见的页面形态是“表单 + 流程”组合：

  • 字段密度高（基础信息、声明项、选择项、上传项）
  • 校验复杂（联动校验、条件必填、格式校验）
  • 流程严格（多步骤、可回退、需留痕）

如果设计与实现不当，会出现：

  • 填写成本高、错误率高
  • 频繁中断后难以恢复
  • 需求迭代带来大量返工

本文总结一套偏工程视角、可复用的表单与流程实践方法。

1. 信息结构：先把“业务阶段”拆清楚

长表单往往不是“字段太多”，而是“阶段混在一起”。推荐按阶段拆分：

  • 基本信息（身份、联系、所属信息）
  • 规则声明（勾选确认、阅读提示）
  • 证明材料（上传、有效期、补充说明）
  • 结果确认（回显、确认提交）

拆分标准：

  • 每一步只解决一类问题
  • 每一步的“完成条件”可清晰定义
  • shows/hides 的动态联动尽量在步骤内完成

2. 多步骤流程：用状态机思维做“可恢复”

对流程页面，最核心的能力是：中断后可恢复。

推荐的最小状态模型

type FlowState = {
  step: number;               // 当前步骤
  completedSteps: number[];   // 已完成步骤
  draft: Record<string, any>; // 草稿数据
  lastSavedAt?: number;       // 最近保存时间
}

保存策略

  • 本地草稿（localStorage/IndexedDB）：快速、无需网络
  • 服务端草稿（可选）：换设备继续、可做留痕

建议组合使用：

  • 输入时本地节流保存（例如 1–2 秒）
  • 步骤完成/离开页面时触发服务端保存（可选）

3. 校验策略：把“校验”当成一个独立系统

常见错误是：校验逻辑散落在各个组件里，最终变成难以维护的条件判断。

推荐做法是：

  • 校验规则集中管理（按字段或按步骤）
  • UI 只负责展示校验结果
  • 支持三种校验时机

三种校验时机

  1. 即时校验：输入完成后触发（格式类最适合）
  2. 离焦校验：失焦触发（减少打断）
  3. 提交校验：步骤提交时触发（联动/条件必填）

示例（伪代码）：

const rules = {
  phone: [
    { when: "blur", test: isPhone, message: "手机号格式不正确" }
  ],
  address: [
    { when: "submit", test: required, message: "联系地址不能为空" }
  ]
}

4. 动态联动：让“条件必填”可解释

金融表单常见“条件必填”：

  • 选择 A -> 需要补充字段 X
  • 勾选声明 -> 才能进入下一步
  • 上传材料 -> 需要填写有效期

工程建议：

  • 联动规则放在“领域层”而不是组件内
  • UI 展示联动原因（为什么这个字段出现/变成必填）

例如在字段旁增加一句辅助说明：

选择“邮寄”后，需要补充收件地址

5. 错误提示：减少“集中爆炸”

用户体验层面，“一提交弹出 10 个错误”通常体验很差。建议：

  • 步骤提交时只聚焦第一个错误，并滚动定位
  • 同时在顶部给出错误概览（列表）
  • 字段级错误提示保持简短、可执行

错误文案建议用“动作 + 原因”结构：

  • ✅ “请补全联系地址”
  • ❌ “联系地址错误”

6. 组件化：把表单能力做成可复用模块

推荐拆成三层：

  • 字段层：Input/Select/Date/Upload 等
  • 表单层：FormContainer（布局、校验、错误聚合）
  • 流程层：FlowContainer（步骤管理、草稿、恢复、提交）

这样做的好处：

  • 字段层专注交互与可访问性
  • 表单层专注校验与展示一致性
  • 流程层专注状态流转与恢复

7. 性能与稳定性：避免“表单卡顿”

高字段密度页面容易卡顿，常见原因：

  • 全表单状态放一个大对象，频繁 setState
  • 每次输入触发大量联动渲染
  • 上传组件与预览渲染过重

建议：

  • 以字段为粒度做局部更新（或使用表单库的订阅机制）
  • 联动计算做 memo/节流
  • 上传预览使用懒加载与压缩策略

8. 安全与隐私：把敏感信息当成默认约束

金融系统常见敏感信息（身份证号、联系方式等）。工程注意点：

  • 页面日志与埋点避免上报原始敏感字段
  • 前端缓存草稿需要做过期策略
  • 截图/复制等能力（若有）需要按场景控制

总结

金融业务系统的表单与流程，难点在于“高约束 + 高复杂度 + 可恢复”。落地时建议优先保障：

  • 信息结构清晰（按阶段拆分）
  • 状态可恢复（草稿与步骤状态）
  • 校验可维护（规则集中、UI 解耦）
  • 联动可解释（减少不透明的必填变化）

当这些基础能力稳定后，流程的新增、调整与复用都会明显更可控。
