一个「销售漏斗 + AI 推荐系统」的架构草图

之前有个一直在脑子里打转的小项目想法：

做一个面向 B 端业务的「销售线索 + AI 推荐」系统， 能够自动挖掘企业线索，给销售每天分配「今天优先联系谁」。

后来花时间把这套东西从「想法」整理成了一个相对完整的架构草图。
这篇就是对那张草图的文字版说明。

1. 业务目标：帮销售做三件事

先不谈技术，先看它要帮销售做什么：

  1. 发现客户

    • 从公开网页、活动、行业名录等地方挖潜在线索
    • 大致知道这些企业做什么、在哪个国家、可能需要什么服务

  2. 管理漏斗

    • 对每一个线索标记当前阶段：
      • 只是数据
      • 已经联系
      • 有意向
      • 报价中
      • 跟进中
      • 成交 / 关闭

  3. 每天给行动建议

    • 不让销售每天打开系统都迷茫：

      「我今天到底该先联系谁？」

    • 而是自动给出一份「今日推荐线索清单」

2. 总体架构：从左到右看一遍

可以大致分成四层：

  1. 数据获取层：爬虫 / 导入 / 手工录入
  2. 数据存储与建模层：线索库 + 客户库 + 行为库
  3. 算法与 AI 层：打分、分配、推荐理由生成
  4. 展示与交互层：Web 后台 + 销售工作台

下面按层展开。

3. 数据获取层：信息从哪里来？

3.1 爬虫 / 抓取

  • 行业展会参展企业名单
  • 物流公司 / 外贸企业黄页
  • 公共招投标信息
  • 同行公司官网上的合作案例

抓取的原则：

  • 只抓 公开网页信息，不触碰敏感或需要登录的数据
  • 遵守爬虫礼仪（限速、尊重 robots.txt 等）
  • 做基础的去重和清洗

3.2 导入 / 手工录入

  • 销售现有 Excel、名片、线下活动名单
  • 手工在系统里新增线索（比如在电话里拿到的联系方式）

这些入口要做得尽量顺手，否则销售不会用。

4. 数据存储与建模层

这里大致有三个核心表：

  1. lead（线索表）
  2. company（公司 / 客户表）
  3. interaction（互动记录表）

简化模型可以是：

company
  - id
  - name
  - country
  - industry
  - website
  - scale
  - tags (数组)

lead
  - id
  - companyId
  - contactName
  - contactEmail
  - contactPhone
  - source (爬虫/导入/手工)
  - stage (初始/已联系/有意向/报价/跟进/成交/关闭)
  - score (系统评分)
  - ownerId (当前负责人)

interaction
  - id
  - leadId
  - type (电话/邮件/面谈/线上会议)
  - content (摘要)
  - result (继续跟进/暂缓/无意向)
  - nextActionTime

所有后续的推荐与分析，都是在这几张表的基础上做文章。

5. 算法与 AI 层：AI 在哪里介入？

可以分成几块：

5.1 线索评分（Scoring）

给每条线索打一个「价值分」：

  • 依据：国家、行业、公司规模、网站内容关键词、历史转化率等
  • 这个可以先从 规则 + 简单模型 做起，不必一上来就上复杂机器学习

AI 在这里可以做的，是：

  • 从公司网站文本里抽取「业务类型」「服务对象」等语义特征
  • 帮助判断这家公司是否符合目标画像

5.2 每日推荐清单

每天给每个销售生成一份：

「今日建议优先处理的 30 条线索」

参考因素包括：

  • 线索评分（越高越优先）
  • 距离上次联系的时间（太久没跟进要提上来）
  • 下一步动作时间（已经约好今天要回电的）
  • 销售的历史成交偏好（比如某销售擅长跟某个国家/行业）

可以先用规则做排序，留一个「AI 调整权重」的空间，逐步学习。

5.3 推荐理由生成

这里可以用大模型做一些「可解释性」增强：

  • 「这家公司近一年在你目标国家有多次发货记录」
  • 「与你最近成交的三家企业业务类型相似」
  • 「距上次沟通已超过 10 天，可以考虑发一次跟进邮件」

这些理由不需要完全精确，但要合理、可信，能帮助销售理解：

「为什么系统推荐让我先联系这些人？」

5.4 沟通内容辅助

再往后一步，可以接入：

  • 帮忙写第一封冷邮件模板
  • 帮忙润色跟进话术
  • 帮忙总结会议纪要，自动写入 interaction.content

这些都可以作为独立的小功能插在工作台内。

6. 展示与交互层：销售每天实际在用什么页面？

我想象里的「销售工作台」大概是这样：

  1. 今日工作列表

    • 顶部卡片：今日推荐数、已处理数
    • 列表：每条线索的公司名、国家、当前阶段、推荐理由

  2. 线索详情页

    • 左侧：公司基本信息 + 历史互动时间线
    • 右侧：AI 提取的关键信息摘要（业务、规模、可能需求等）
    • 下方：新增互动记录的表单（含 AI 生成备注草稿的按钮）

  3. 漏斗视图

    • 以阶段为列，展示当前各阶段线索数量
    • 可以拖拽线索在不同阶段之间移动（状态变化也会被记录）

7. 风险与现实约束

这种系统在真实世界里有不少约束：

  1. 合规与隐私

    • 只能使用公开信息
    • 联系信息的获取与使用必须符合当地法律

  2. 数据质量

    • 爬虫来的数据一定很脏，要有完整的清洗、去重、合并逻辑
    • 销售手工录入的信息更是五花八门

  3. 团队接受度

    • 如果推荐列表经常「不靠谱」，销售很快就会失去信任
    • 前期可以只做「建议」，不做硬性 KPI 绑定

  4. AI 并不能取代销售的判断

    • 能做的是帮忙缩小「选择范围」
    • 最后联系谁、怎么聊，仍然要靠人来决定

8. 小结

这个「销售漏斗 + AI 推荐系统」的架构草图，大概可以概括成：

  1. 用爬虫和导入，把线索数据尽可能收集全
  2. 建一个结构合理的线索 + 公司 + 互动记录模型
  3. 用规则 + 简单算法 + AI，给线索打分和生成每日推荐清单
  4. 在销售工作台上，用「今日推荐 + 推荐理由」的方式呈现
  5. 在此基础上再逐步加上话术辅助、纪要生成等 AI 小功能

它更像是一个长期演进的系统，而不是一次性做完的产品：
先把「让销售每天知道该干什么」这件事做好，其它的可以慢慢加。
