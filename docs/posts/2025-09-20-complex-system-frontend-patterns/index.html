复杂业务系统前端的通用设计模式

背景

复杂业务系统的前端常见特征：

  • 页面多、模块多、角色多
  • 流程长、状态复杂、联动规则多
  • 需求迭代频繁，历史包袱逐步累积

这类系统的前端设计，核心目标是：用结构化手段降低不确定性。下面总结一组可迁移的通用模式。

模式 1：流程驱动页面结构（Flow-Driven UI）

当页面属于“多步骤流程”时，建议将流程当成第一等公民：

  • 流程配置（步骤定义、进入条件、完成条件）
  • 步骤模块（每步独立校验与提交）
  • 流程控制器（负责跳转、恢复、草稿、回退）

结构示意：

flow/
  config.ts
  controller.ts
  steps/
    stepA.tsx
    stepB.tsx

收益：

  • 步骤增删更可控
  • 单步问题定位更快
  • 流程可复用（不同业务只换配置/步骤）

模式 2：状态集中 + 边界清晰（State Boundary）

状态混乱通常来自“该共享的不共享、不该共享的全局化”。

建议将状态分层：

  • 全局：会话、权限、字典、全局配置
  • 模块：业务模块内共享状态（列表筛选、分页、缓存）
  • 页面：页面局部状态（输入、展开、临时选中）

并明确“谁拥有数据的真相”（Single Source of Truth），减少多处重复维护。

模式 3：规则外置（Rules as Data）

复杂系统最怕“规则散落在 UI 代码里”。推荐将规则外置成数据：

  • 条件必填
  • 展示/隐藏
  • 权限控制
  • 校验提示

示例（伪代码）：

{
  "field": "address",
  "requiredWhen": "deliveryMode == 'mail'",
  "message": "选择邮寄后需要填写收件地址"
}

收益：

  • 规则可审查（code review 更容易）
  • 规则可复用（多页面共享）
  • UI 更干净（减少 if/else 地狱）

模式 4：公共能力组件化（Capability Components）

复杂系统中常见的“公共能力”包括：

  • 查询面板（筛选、重置、保存条件）
  • 表格（分页、排序、列配置、导出）
  • 表单（校验、错误聚合、联动）
  • 弹窗（确认、危险操作二次确认）

建议以“能力”为中心抽象组件，而不是以“页面”为中心复制粘贴。

收益：

  • 一致性提升
  • 维护成本下降
  • 新页面交付速度更稳定

模式 5：可观测性内建（Observability by Default）

复杂系统的问题定位成本很高，应把观测能力作为默认能力：

  • 错误：异常、资源失败、接口失败
  • 性能：首屏、切换、长任务
  • 业务：关键流程成功率、失败原因分布

关键原则：

  • 日志脱敏（默认不记录敏感字段）
  • 链路可追踪（traceId 打通前后端）
  • 聚合去重（避免噪声告警）

模式 6：发布治理（Feature Flag + 灰度）

复杂系统常常需要“边迭代边运行”。发布治理建议具备：

  • 功能开关：按角色/比例开启
  • 灰度：先小范围验证指标再扩大
  • 回滚：资源与配置可回退
  • 兼容：新旧接口/新旧字段并存一段时间

总结

复杂业务系统前端的竞争力，往往不在“写得快”，而在“长期可维护、可演进”：

  • 流程驱动结构
  • 状态边界清晰
  • 规则外置
  • 公共能力组件化
  • 可观测性与发布治理内建

这些模式与具体框架无关，适用于多数中大型业务系统的长期演进。
